其实, 我们应该清楚, js 是一门基于浏览器运行的语言, 它最初的环境就是浏览器, 那么它的一些功能就和浏览器其实是分不开的.

#### 事件循环

既然我们要去探讨事件循环, 那么我们就应该知道它是什么`event loop`, 是指在浏览器的 `js 引擎`中等待任务, 执行任务, 进入休眠状态等待更多任务这几个状态之间转换的无限循环。
也就是下图中的`event loop`, 你可以将它理解成一个算法。这个算法要完成的功能就是将上面的任务完成
![](https://pic4.zhimg.com/80/v2-da078fa3eadf3db4bf455904ae06f84b_720w.jpg)
那么，我们这里提到了`js引擎`, 并且提到了事件循环是其中的一部分, 那么`js引擎`到底干什么的呢?
其实它的全称应该为`js引擎线程`(js 内核), 它的功能主要为: 解析和执行 JavaScript 脚本。
它在大多数的时候都是不进行任何操作的, 它只有在脚本处理程序事件激活的时候执行。
那么又有一个问题, 我说了这么多`js引擎线程`, 那么`event loop`就是由它控制的吗? 其实不是的, 它是由`事件触发线程`去控制的. 那么事件触发线程是如何去控制`event loop`的呢？这个后面去讲.
那么现在读到这里, 有人就说了, 停你快拉到吧, 啥进程线程的, 去你的吧.
不急, 我们现在来看一下浏览器中的进程和线程!

#### 浏览器中的进程和线程

我们电脑有一个 cpu, 它负责计算机的"计算"任务,它很强大,但是它一个时间只能去干一件事。++进程++ 就是 cpu 能处理的最小的单个任务,每个应用程序就是一个进程。 它使用++时间片轮转进度算法++来实现同时运行多个进程......这不重要(我们主要去看浏览器), 浏览器也是其中的一个进程, 它有一个主进程还有很多的子进程, 通过子进程来实现多进程。

:::primary
当我们打开浏览器的时候, 我们的浏览器就会默认的给我们打开++至少++ **1 个浏览器进程**, **1 个 GPU 进程**, **1 个网络进程**, 和**1 个渲染进程**(一个 tag 页就是一个渲染进程)。ps:\*如果你有浏览器的第三方插件也会打开**第三方插件进程\***
然后其中, 浏览器进程, GPU 进程,网络进程,第三方插件进程这些都是共有的, 只有渲染进程会跟着 tag 页的增加而增加。这里的渲染进程还有另外一个名字, 就是我们常说的浏览器内核！！！
:::
这里以 chrome 浏览器举例去看我们的进程, 设置 ==> 更多工具 ==> 任务管理器
![](https://tva4.sinaimg.cn/large/006xVoHRly8h072xctzi3j30w00iydj5.jpg)
进程的下一层就是线程, 我们常说的**js 引擎线程**就是在渲染进程中,那么我们的渲染进程中到底包含什么线程呢?

- GUI 渲染线程: 负责渲染页面, 解析 HTML 和 CSS, 构建 DOM 树, CSSOM 树, 渲染树和绘制页面, 重绘重排也是在该线程中执行.
- js 引擎线程: 一个渲染进程中只能有一个 js 引擎线程,负责解析和执行 js.
- 计时器线程: 指 setInterval 和 setTimeout，因为 JS 引擎是单线程的，所以如果处于阻塞状态，那么计时器就会不准了，所以需要单独的线程来**负责计时器工作**
- 异步 http 请求线程: XMLHttpRequest 连接后浏览器开的一个线程，比如请求有回调函数，异步线程就会将回调函数加入事件队列，等待 JS 引擎空闲执行
- 事件触发线程: 主要用来控制事件循环，比如 JS 执行遇到计时器，AJAX 异步请求等，就会将对应任务添加到事件触发线程中，在对应事件符合触发条件触发时，就把事件添加到待处理队列的队尾，等 JS 引擎处理

上面我们大概的讲了一下，浏览器中的进程和线程, 那么大家对开头讲的内容应该是有了一些深入的理解。那么就让我们回到事件循环

#### 事件循环细读

我们需要知道事件循环到底是为了什么? 我们既然有 js 引擎去解读 js 代码, 那么就让他去执行呗。那么现在来想一个场景::::> 当你在 js 代码中写入一个 http 请求,写入一个触发事件， 这个事件需要延迟 30 秒执行（可以有些夸张）, 那么当这些长耗时的代码执行的时候，我们的页面就陷入了一种假死的状态, 用户又不会知道我们在干什么！所以, 为了把这些长耗时的操作提出来, 大佬们研究了异步机制， 于是事件循环机制也就应运而生。这也就是 js 虽然**单线程**但是却**非阻塞**。
那么事件循环的工作环境是什么样的呢？
现在回到我们当时的第一张图
![](https://pic4.zhimg.com/80/v2-da078fa3eadf3db4bf455904ae06f84b_720w.jpg)
我们可以发现,其中 `event loop`只是其中的一部分, 其中还有 heap(堆), stack(栈), callback queue(回调事件队列), WebAPIs. 那么这些都是什么呢?
js 引擎在执行 js 代码的时候, 会有一个执行栈, js 代码按先后顺序入栈和执行结束后出栈.
会有一个由`事件触发线程`维护的回调事件队列,
有专门处理异步事件的 webapi 接口,这些接口由渲染进程的线程提供.毕竟总不可能让你之间用 js 代码去人家线程上瞎玩吧。玩坏了这个 tag 页就 over 了。也很符合程序的封装

##### 何谓任务？

只要你去了解过事件循环, 那么你就一点会知道一个词叫 `任务队列`, 还会分化出来`宏任务队列`, `微任务队列`。那么抛去 宏任务(macro task)， 微任务(micro task)来说, 什么是任务？

结合我们的日常生活的话,我们一时间做不完打算后面再去做的才叫任务, 那么延伸一下, 是不是我们的任务列表就可以对照到任务队列？
当 js 引擎在处理`script`整体代码的时候, 我们移动了鼠标, 点击了键盘,一个计时任务结束, 那么我们这时候任务就会被存入一个队列标明他们的先后顺序，到时候让 js 引擎去按顺序渲染.这个队列就被叫做**任务队列**即宏任务队列,其实一开始是没有宏任务和微任务之分的, 所以之前提的任务队列是指宏任务队列，现在是指宏微任务。那么两者都有什么区别呢? 为什么要再弄一个微任务出来?

##### 宏任务和微任务

稍微了解的话，我们都会知道, 宏任务和微任务都有很多（假装很多,只是量词）类型的事件组成。那么我们先不说他们由什么组成. 我们先给出他们的一个含义（非官方）
https://www.zhihu.com/question/316514618 （有个回答讲的很好, 我现在水平还不能完全吸收）
:::primary
宏任务: 多线程的异步协作
微任务: js 语言的本地异步(延迟执行)
:::
那么现在去看, 多线程是哪几个线程, <a href="https://www.bilibili.com/video/BV1z34y1m7La?spm_id_from=333.337.search-card.all.click">又回到最初的起点</a> !!请大家听歌!!
我们可以看到我们一直都没有提到`异步http请求线程`和`定时触发线程`, 那么我们这里就要讲这个了。
先看一个图
![](https://p9-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/95d80f4ce96e4e039b051c048e8630f3~tplv-k3u1fbpfcp-zoom-in-crop-mark:1304:0:0:0.awebp)
上面的话，我们可以看到， js 引擎在遇到 **定时器**, **http 请求**, 就分别交给对应线程去执行。然后等执行结果出现后再讲对应的回调函数交给事件触发线程，由它放入事件队列。
所以, 我们可以知道定时器, http 请求都是异步的宏任务。

那微任务呢? 微任务是后面加进来的,这里我们就暂时知道 promise 是微任务就行(这个不是重点), 微任务我们上面说是 js 语言的本地异步(延迟执行),那么它到底是那一段 js 的延迟执行呢?
其实它是生成它的宏任务的延迟执行, 所以在执行完一个宏任务后要去清除微任务队列中全部的微任务。并且这样也有一个好处就是，将微任务现在在自己的作用域中, 防止对其他的异步任务的干扰
